# 2. Архітектура проєкту

## 2.1. Принцип розбиття на застосунки

Проєкт розбито за доменними межами, з урахуванням того, що продукт значно простіший за типову UGC-платформу — немає модерації контенту від користувачів, немає голосування. Основна складність зосереджена в логіці генерації орнаменту і в редакційній достовірності культурного контенту.

| Застосунок | Зона відповідальності                                                                                                 | Ключові сутності                                           |
| ---------- | --------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| `core`     | Спільне: абстрактні моделі, утиліти, context processors, sitemaps, обробники помилок, кастомні теги                   | базові абстрактні моделі                                   |
| `accounts` | Профілі користувачів, кастомізація Allauth, налаштування                                                              | профіль користувача                                        |
| `patterns` | Серце продукту: регіони, мотиви, джерела достовірності, щоденні патерни, генеративний алгоритм, колекції користувачів | регіон, мотив, джерело, щоденний патерн, збережений патерн |
| `blog`     | Двомовний блог із WYSIWYG, категорії статей, підтримка гостьових/спонсорованих матеріалів                             | стаття, категорія статті, заявка на гостьовий пост         |
| `pages`    | Статичні сторінки, FAQ, контактна форма                                                                               | статична сторінка, пункт FAQ, контактне повідомлення       |

**Свідомо відхилено:** окремий застосунок для генерації орнаменту як мікросервісу — для одного розробника це надмірне ускладнення. Алгоритм генерації живе як внутрішній модуль (сервісний шар) усередині `patterns`. Якщо логіка генерації переросте приблизний обсяг у кілька сотень рядків — виноситься в окремий підпакет `patterns/generation/`, залишаючись усередині того самого застосунку.

## 2.2. Діаграма архітектури (текстова)

Клієнт (браузер, Bootstrap 5 + мінімальний Vanilla JS для інтерактивних елементів колекції) звертається через HTTPS до реверс-проксі (на фазі PythonAnywhere — вбудований проксі платформи; на фазі власного сервера — Nginx). Django-застосунок обробляє запит через WSGI-сервер (вбудований сервер платформи на PythonAnywhere; Gunicorn на власному сервері). Усередині Django — застосунки `pages`, `blog`, `accounts` спираються на `patterns` як центральний домен, а всі — на `core` як фундамент. База даних — SQLite на старті (файл на диску хостингу), з можливістю переходу на зовнішній керований PostgreSQL без зміни коду (через абстракцію рядка підключення). Немає залежності від будь-якого зовнішнього API — вся генерація контенту відбувається локально.

**Напрямок залежностей** (правило: усі дивляться вниз, ніхто не імпортує з вищого рівня):

`core` — фундамент, ні від кого не залежить. `accounts` залежить від `core`. `patterns` залежить від `core` і `accounts` (через зв'язок збереженого патерну з користувачем). `blog` і `pages` залежать лише від `core`. Крос-доменні посилання (наприклад, стаття блогу що згадує конкретний патерн) реалізуються через slug/URL, не через прямий зовнішній ключ між застосунками.

## 2.3. Структура папок і файлів

Кореневий рівень містить стандартний набір: файл керування, конфігурацію інструментів якості коду, файли опису залежностей (розбиті на базові, локальні для розробки, продакшн), приклад файлу змінних середовища, ігнор-файли для Git і Docker, конфігурацію pre-commit хуків, конфігурацію тестового раннера, файли Docker для локальної розробки, workflow-файл CI/CD.

Проєктний пакет конфігурації (не застосунок, а інфраструктурний пакет) містить розбиті за середовищем налаштування (базові, локальні, продакшн, тестові), кореневий роутинг з підтримкою мультимовності, точки входу WSGI/ASGI.

Директорія застосунків містить кожен із перелічених вище доменних застосунків зі стандартною внутрішньою структурою: моделі, форми, представлення, сервісний шар для нетривіальної бізнес-логіки, шар вибірок для складних читань, сигнали, реєстрація перекладів, реєстрація в адміністративній панелі.

Окремо — директорія локалізаційних файлів (українська, англійська), директорія вихідних статичних файлів (стилі, скрипти, зображення бренду — не зображення патернів, бо вони не зберігаються як файли), директорія зібраної статики (не в контролі версій), директорія шаблонів із глобальними частковими шаблонами (шапка, підвал, перемикач мови, картка патерну, пагінація, повідомлення) і піддиректоріями за кожним застосунком, директорія для початкових даних наповнення (фікстури регіонів і мотивів для першого розгортання).

**Ключові архітектурні рішення:**

- Проєктний конфігураційний пакет відокремлений від назви продукту — полегшує перейменування чи білий лейбл у майбутньому.
- Налаштування розбиті на середовища без умовних розгалужень у коді — вибір середовища через змінну середовища, що вказує на конкретний модуль налаштувань.
- Сервісний та вибірковий шари відокремлюють бізнес-логіку від тонких представлень — критично важливо для алгоритму генерації орнаменту, який має бути незалежно тестованим поза контекстом HTTP-запиту.

# 3. Моделі бази даних

## 3.1. Абстрактні моделі (застосунок `core`)

**Модель часових позначок** (абстрактна): поле часу створення (автоматичне заповнення при створенні, індексоване — за ним сортується більшість списків) і поле часу останнього оновлення (автоматичне оновлення при кожному збереженні).

**Модель ЧПУ-ідентифікатора** (абстрактна): поле slug (унікальне, індексоване, генерується автоматично при першому збереженні, якщо не задане вручну). Правило: slug завжди складається виключно з латинських символів незалежно від мови інтерфейсу — генерується один раз через транслітерацію кириличного заголовка (бібліотека транслітерації з явним вимкненням юнікод-символів у результаті, щоб уникнути потворного відсоткового кодування в URL). Мовний префікс URL (наявність чи відсутність `/en/`) — відповідальність маршрутизатора, не самого slug-поля; один і той самий slug обслуговує обидві мовні версії однієї сутності.

## 3.2. `Region` (регіон) — застосунок `patterns`

Ключова довідкова сутність, що визначає палітру й символіку кожного дня. Перекладна модель (назва і опис символіки — окремі поля для кожної мови).

| Поле                   | Призначення                                                                                                                                                                       |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Назва регіону          | перекладна, наприклад «Полтавщина» / «Poltava region»                                                                                                                             |
| Опис символіки         | перекладний, розгорнутий текст (кілька речень) про значення кольорів і мотивів саме цього регіону — використовується і на сторінці регіону, і як короткий підпис під патерном дня |
| Домінантні кольори     | структуроване поле (список кольорових кодів), що визначає палітру генерації для цього регіону                                                                                     |
| Тип крою сорочки       | текстове поле, що описує особливості крою (для майбутнього розширення — наразі інформаційне)                                                                                      |
| Порядок у ротації      | ціле число, що визначає позицію регіону в календарній послідовності                                                                                                               |
| Активний               | булеве поле, що дозволяє тимчасово виключати регіон з ротації без видалення даних                                                                                                 |
| **Джерела**            | зв'язок «багато-до-багатьох» із моделлю джерела (розділ 4) — обов'язкове поле, порожній перелік джерел блокує публікацію (розділ 4.4)                                             |
| **Статус верифікації** | вибір: «на перевірці» / «верифіковано» / «потребує перегляду» — службове поле для редакційного процесу, детально в розділі 4                                                      |
| Slug                   | від абстрактної моделі ЧПУ-ідентифікатора                                                                                                                                         |

Slug регіону не перекладається — та сама причина, що й у інших проєктах: уникнення дублювання canonical-адрес.

## 3.3. `Motif` (мотив) — застосунок `patterns`

Довідник окремих геометричних елементів, з яких композується орнамент. Перекладна модель.

| Поле                   | Призначення                                                                                                                                                                          |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Назва мотиву           | перекладна, наприклад «Ромб», «Барвінок» (обережно з символами, що мають неоднозначні чи політично чутливі конотації — курована перевірка на етапі наповнення довідника, розділ 4.5) |
| Опис значення          | перекладний, коротке пояснення що символізує мотив                                                                                                                                   |
| Параметри геометрії    | структуроване поле, що описує як мотив будується алгоритмічно (базові координати, правила симетрії, які саме групи симетрії застосовні)                                              |
| Сумісні регіони        | зв'язок «багато-до-багатьох» із моделлю регіону — які мотиви типові для яких регіонів                                                                                                |
| **Джерела**            | зв'язок «багато-до-багатьох» із моделлю джерела — той самий принцип, що й у регіону                                                                                                  |
| **Статус верифікації** | той самий принцип, що й у регіону                                                                                                                                                    |
| Активний               | булеве поле                                                                                                                                                                          |

## 3.5. `DailyPattern` (щоденний патерн) — центральна модель

| Поле                    | Призначення                                                                                                                                                                                                                                       |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Дата                    | унікальне поле дати — один патерн на календарний день, індексоване                                                                                                                                                                                |
| Регіон                  | зовнішній ключ на модель регіону, заборона видалення регіону поки є пов'язані патерни                                                                                                                                                             |
| Seed генерації          | ціле число чи рядок, детермінований на основі дати — гарантує що повторний запуск алгоритму на ту саму дату дає ідентичний результат                                                                                                              |
| Версія алгоритму        | ціле число — критично важливе поле для відтворюваності: якщо алгоритм генерації колись зміниться, історичні патерни повинні залишатись прив'язаними до версії алгоритму, якою вони були створені вперше, а не перегенеровуватись за новою логікою |
| Згенерований SVG        | текстове поле, що зберігає готовий результат генерації (не файл зображення — текстовий XML-вміст SVG)                                                                                                                                             |
| Використані мотиви      | зв'язок «багато-до-багатьох» із моделлю мотиву — які саме мотиви увійшли до конкретного патерну дня, для аналітики й майбутніх фільтрів                                                                                                           |
| **Статус генерації**    | вибір: «успішно» / «помилка, застосовано резервний варіант» — службове поле для діагностики (розділ 8.6)                                                                                                                                          |
| Лічильник переглядів    | ціле число, не індексується                                                                                                                                                                                                                       |
| Час створення/оновлення | від абстрактної моделі часових позначок                                                                                                                                                                                                           |

**Ключове архітектурне рішення:** генерований SVG зберігається в базі даних як текст (не файл на диску), тому що: по-перше, SVG — текстовий формат і важить кілобайти, не мегабайти; по-друге, зберігання в базі спрощує бекап (текстові дані бекапляться разом з рештою бази без окремої стратегії для файлового сховища); по-третє, знімає будь-яку залежність від файлової системи хостингу, що важливо саме для обмеженого дискового простору на безкоштовному плані.

## 3.6. `SavedPattern` (збережений патерн користувача) — колекція

| Поле           | Призначення                                 |
| -------------- | ------------------------------------------- |
| Користувач     | зовнішній ключ на модель користувача        |
| Патерн         | зовнішній ключ на конкретний `DailyPattern` |
| Час збереження | від абстрактної моделі часових позначок     |

Обмеження унікальності на парі «користувач + патерн» — неможливо зберегти той самий патерн двічі.

## 3.7. `Profile` (профіль користувача) — застосунок `accounts`

Розширення стандартної моделі користувача через зв'язок «один-до-одного».

| Поле                             | Призначення                                                                             |
| -------------------------------- | --------------------------------------------------------------------------------------- |
| Користувач                       | зв'язок «один-до-одного»                                                                |
| Відображуване ім'я               | необов'язкове, порожнє значення означає використання імені користувача за замовчуванням |
| Мова інтерфейсу за замовчуванням | необов'язкове поле, дублює вибір, збережений у сесії/cookie                             |

**Сигнал:** автоматичне створення профілю при створенні нового користувача, з обов'язковою перевіркою прапорця «сирого» збереження, щоб уникнути помилки цілісності при повторному створенні через завантаження фікстур.

|                                             | **`Source`** (бібліографічні дані не перекладаються — джерело існує мовою публікації) |

## 3.12. Стратегія міграцій

**Порядок створення:** базовий застосунок не створює власних таблиць. Застосунок профілів — модель профілю. Застосунок патернів — спочатку модель джерела (незалежна), потім довідники регіону й мотиву (що посилаються на джерела), потім центральна модель щоденного патерну, потім модель збереження. Застосунки блогу й сторінок — незалежні один від одного.

**Реєстрація перекладних полів** — виконана до першої міграції даних.

**Правило безпечних змін схеми без даунтайму:** додавання необов'язкового поля — без ризику; перетворення поля на обов'язкове — трьома кроками; перейменування поля — додати нове, скопіювати, підтримувати обидві версії коду протягом одного релізу, видалити старе окремо; додавання індексу на велику таблицю — без блокування читання/запису.

**Початкові дані:** окремий набір фікстур чи керована команда для завантаження першого набору регіонів і мотивів разом із джерелами — необхідний крок перед першим продакшн-розгортанням (деталі в розділі 18.5, чек-лист перед запуском).

# 4. Достовірність контенту та джерела — редакційна методологія

> Цей розділ — прямий наслідок вимоги: платформа працює з культурною спадщиною, тому жодне твердження про символіку не публікується без перевіреної основи.

## 4.1. Проблема, яку свідомо вирішуємо

Значна частина інтернет-контенту про «значення символів вишиванки» — романтизована чи відверто вигадана: псевдоязичницькі трактування, що поширились через соцмережі й не мають підтвердження в академічній етнографії. Продукт, що претендує на культурну достовірність, не може некритично відтворювати такий контент.

## 4.2. Ієрархія довіри до джерел

| Рівень довіри                             | Тип джерела                                                                                               | Приклади категорій                                                                                                                               |
| ----------------------------------------- | --------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| Найвищий                                  | Академічні етнографічні дослідження, опубліковані наукові праці про символіку й техніку вишивки           | монографії, статті в фахових журналах, дисертаційні дослідження                                                                                  |
| Високий                                   | Музейні колекції та їхні офіційні каталоги/публікації                                                     | етнографічні музеї, музеї народного декоративного мистецтва                                                                                      |
| Високий                                   | Видання державних культурних інституцій                                                                   | міністерство культури, інститути народної творчості                                                                                              |
| Середній, з явним маркуванням             | Регіональна усна традиція, зафіксована в польових етнографічних записах                                   | локальні краєзнавчі джерела, записані перекази — публікуються з explicit позначкою «за регіональною традицією», не як встановлений науковий факт |
| Не використовується як самостійне джерело | Неатрибутовані інтернет-статті, блоги без покликань, контент, згенерований мовними моделями без перевірки | —                                                                                                                                                |

## 4.3. Правило розрізнення задокументованого і фольклорного

Кожен опис символіки на сторінці регіону чи мотиву явно позначає, якого типу твердження наводиться: **задокументоване** (спирається на джерело найвищого чи високого рівня довіри) чи **за усною традицією** (регіональний переказ, чесно позначений як такий, без видавання за наукового консенсусу). Змішування цих категорій без розрізнення — заборонено редакційним правилом.

## 4.4. Обов'язковість джерела перед публікацією

Технічно модель `Region` і `Motif` мають обов'язковий зв'язок із хоча б одним записом `Source`. Редакційний робочий процес: запис регіону чи мотиву переводиться зі статусу «на перевірці» у статус «верифіковано» лише після того, як додано хоча б одне джерело з рівнем довіри «високий» чи «найвищий», або явно позначено як «за усною традицією» з поясненням у нотатці редактора. Публічна видимість на сайті прив'язується до статусу верифікації — неверифікований запис не потрапляє в активну ротацію генерації.

## 4.5. Особлива обережність із чутливими символами

Деякі геометричні мотиви традиційної вишивки історично мають неоднозначне сучасне сприйняття (через випадковий геометричний збіг з символами, що набули іншого, небажаного значення в XX столітті). Редакційне правило: перед додаванням будь-якого мотиву до активного довідника — явна перевірка на такі збіги, і за наявності ризику — або свідома відмова від мотиву, або розгорнутий пояснювальний контекст на сторінці мотиву, що напряму розводить автентичне значення від небажаної асоціації.

## 4.6. Джерела для статей блогу

Статті, що містять історичні чи символічні твердження (а не суто інформаційні пости на кшталт оголошень), також пов'язуються з моделлю `Source`. Для гостьових і спонсорованих матеріалів — джерела перевіряються редактором сайту нарівні з редакційними статтями; партнерський статус не звільняє від вимоги достовірності.

## 4.7. Прозорість для читача

На сторінці кожного регіону — видимий читачеві блок «Джерела» з переліком використаної літератури чи колекцій (не лише внутрішня службова інформація для редактора, а публічна демонстрація сумлінності). Це одночасно й етичний стандарт, і природний елемент довіри до бренду, і додатковий текстовий контент, корисний для SEO.

## 4.8. Періодичний перегляд

Записи зі статусом «потребує перегляду» (позначаються вручну, якщо з'являється нова інформація, яка ставить під сумнів раніше внесений опис) підлягають пріоритетній ревізії перед черговим використанням цього регіону в ротації генерації.

# 5. Повний опис кожної сторінки

## 5.1. Головна сторінка (`/`)

**Призначення:** головний рітуальний елемент продукту. Показує силует сорочки з накладеним орнаментом сьогоднішнього дня.

**Логіка отримання даних:** контролер перевіряє, чи існує запис `DailyPattern` на сьогоднішню календарну дату. Якщо існує — використовує збережений SVG. Якщо не існує — викликає сервіс генерації (розділ 8), зберігає результат, потім рендерить. Якщо генерація завершується помилкою — застосовується резервний сценарій (розділ 8.6).

**Блоки сторінки:**

- Великий силует сорочки з накладеним орнаментом по центру.
- Назва регіону дня та короткий уривок опису символіки, з видимою позначкою типу твердження (задокументоване / за традицією).
- Кнопка «Зберегти в колекцію» для авторизованих користувачів; для анонімних — веде на сторінку входу зі збереженням наміру повернутися після авторизації.
- Стрічка-календар останніх кількох днів із мініатюрами для швидкого переходу в архів.
- Короткий блок із останніми статтями блогу.

**Доступність:** SVG-елемент супроводжується текстовим описовим атрибутом (короткий опис «орнамент [назва регіону], дата»), щоб контент був осмисленим для читачів екрана, а не «порожньою графікою».

**Кешування:** фрагмент із самим SVG-патерном кешується на добу з інвалідацією за ключем дати. Персональний стан не кешується разом із загальним HTML.

**Права доступу:** доступно всім. Кнопка збереження активна лише для авторизованих.

**Крайові випадки:** перший запуск проєкту без жодного попереднього патерну — генерація відбувається одразу при першому відвідуванні.

## 5.2. Архів патернів (`/archive/`)

**Призначення:** перегляд усіх попередніх днів з можливістю фільтрації.

**Блоки:** сітка мініатюр патернів у зворотному хронологічному порядку; панель фільтрів за регіоном і діапазоном дат; пагінація.

**Логіка вибірки:** базовий запит по всіх існуючих записах `DailyPattern`, з попереднім завантаженням пов'язаного регіону.

**Права доступу:** усі.

**Крайові випадки:** фільтр за неіснуючим/неактивним регіоном — порожній стан із поясненням.

## 5.3. Сторінка конкретного патерну (`/pattern/<дата>/`)

**Призначення:** окрема сторінка одного дня — для глибокого посилання, поширення в соцмережах, індексації пошуковиками.

**Блоки:** великий SVG-патерн, повний опис символіки регіону цього дня з блоком джерел (розділ 4.7), перелік мотивів що увійшли до композиції з коротким поясненням кожного, кнопка збереження, навігація «попередній день / наступний день», OG-теги для коректного прев'ю при поширенні.

**Права доступу:** усі.

**Крайові випадки:** запит майбутньої дати (теоретично неможливий за логікою лінивої генерації, захист про всяк випадок) — сторінка не знайдена, не помилка сервера.

## 5.4. Сторінка регіону (`/regions/<slug>/`)

**Призначення:** окрема освітня сторінка про кожен регіон — головний SEO-актив продукту.

**Блоки:** назва регіону, повний опис символіки (з розрізненням задокументованого й традиційного), палітра кольорів візуально, галерея мініатюр усіх патернів що коли-небудь генерувались для цього регіону, публічний блок джерел, посилання на пов'язані статті блогу.

**Права доступу:** усі.

## 5.5. Моя колекція (`/collection/`)

**Призначення:** особистий перелік збережених користувачем патернів.

**Логіка вибірки:** усі записи `SavedPattern` поточного користувача з попереднім завантаженням пов'язаного патерну й регіону.

**Додатковий елемент:** індикатор «пройденого туру» — скільки з усіх активних регіонів користувач уже зберіг хоча б один патерн.

**Права доступу:** лише авторизований власник.

**Крайові випадки:** порожня колекція — заохочувальний порожній стан.

## 5.6. Налаштування профілю (`/profile/settings/`)

**Призначення:** мінімальне редагування власного профілю.

**Права доступу:** лише власник, об'єкт для редагування завжди визначається з поточного авторизованого користувача, ніколи з параметра адреси.

## 5.7. Блог — список (`/blog/`)

**Призначення:** список опублікованих статей.

**Блоки:** сітка карток статей, фільтр за категорією, пагінація.

**Права доступу:** усі бачать лише опубліковане; чернетки — виключно персонал через адміністративну панель.

## 5.8. Блог — стаття (`/blog/<slug>/`)

**Блоки:** обкладинка, заголовок, метадані, явна позначка «Партнерський матеріал» для спонсорованого/гостьового типу, санітизований HTML-вміст, блок джерел (для статей що містять історичні твердження), структуровані дані, блок «читайте також», хлібні крихти.

**Крайові випадки:** неіснуючий slug чи чернетка для не-персоналу — сторінка не знайдена.

## 5.9. Сторінка «Про нас» (`/about/`)

Стандартна статична сторінка через модель `StaticPage` із зарезервованим slug.

## 5.10. FAQ (`/faq/`)

Акордеон запитань, згрупованих за категорією.

## 5.11. Контакти (`/contacts/`)

Контактна форма з honeypot-перевіркою і обмеженням частоти.

## 5.12. Заявка на гостьовий пост (`/blog/propose/`)

**Призначення:** публічна форма для зовнішніх авторів/брендів, що хочуть запропонувати матеріал.

**Логіка обробки:** збереження заявки в `GuestPostSubmission`; сповіщення адміністратору (синхронно на етапі PythonAnywhere Free).

**Права доступу:** усі можуть подати; перегляд і розгляд — лише через адміністративну панель.

## 5.13. Вхід, реєстрація, відновлення паролю

Стандартний потік: вхід через електронну пошту й пароль або через Google, обов'язкове підтвердження електронної пошти, повний цикл відновлення паролю, захист від перебору паролів.

## 5.14. Сторінки помилок (404, 500)

Дружні, мінімалістичні сторінки без залежності від бази даних у випадку 500.

## 5.15. Підвал сайту

Колонка бренду з описом і посиланнями на соціальні мережі; колонка навігації (архів, моя колекція, блог); колонка регіонів (кілька найпопулярніших як прямі посилання); колонка компанії (про нас, FAQ, контакти, форма пропозиції гостьового поста); колонка правових документів; нижній рядок з копірайтом і перемикачем мови.

**Логіка видимості без JavaScript:** тіло сторінки — гнучкий контейнер на всю висоту вікна перегляду, основний вміст розширюється, підвал природно опиняється внизу вмісту на довгих сторінках і притискається до низу вікна на коротких.

# 6. Форми і валідація

## 6.1. Спільні принципи

Форми, пов'язані з моделями — автоматична генерація полів з моделі; форми без прямого зв'язку з моделлю — довільні форми. Помилки відображаються стандартизовано. Захист від міжсайтової підробки запиту — токен у кожній формі. Обмеження частоти подання — на рівні обробника форми.

## 6.2. Форма налаштувань профілю

Поле відображуваного імені з обмеженням довжини.

## 6.3. Контактна форма

Поля імені, email, теми (необов'язкова), тексту повідомлення (з мінімальною довжиною), прихована honeypot-перевірка. Обмеження частоти — кілька подань на годину з однієї IP-адреси.

## 6.4. Форма заявки на гостьовий пост

Аналогічна структура, з полями назви бренду й теми пропозиції. Суворіше обмеження частоти подання.

## 6.5. Форми автентифікації

Стандартні форми бібліотеки Allauth, кастомізовані мінімально.

# 8. Система генерації орнаменту — центральна унікальна логіка

## 8.1. Загальний принцип

Генерація повністю детермінована: та сама дата з тим самим регіоном і тією самою версією алгоритму завжди дає ідентичний результат.

## 8.2. Джерело «зерна» генерації

Календарна дата перетворюється на числове значення, яке слугує початковим станом генератора псевдовипадкових чисел. Усі подальші випадкові рішення алгоритму виводяться саме з цього початкового стану — відтворювані, не справді випадкові.

## 8.3. Роль регіону в ротації

Регіони проходять по колу в календарному порядку, визначеному полем порядку в моделі регіону. Якщо регіонів менше, ніж днів наперед, послідовність повторюється циклічно.

**Крайовий випадок — додавання нового регіону посеред циклу:** новий регіон, доданий адміністратором, отримує порядковий номер у кінці поточної послідовності активних регіонів за замовчуванням (не порушує вже заплановану близьку послідовність днів, що вже відбулися чи неминуче настануть найближчим часом); адміністратор може вручну змінити позицію, якщо потрібно включити регіон раніше, з явним розумінням, що це змістить дні для регіонів, які йшли після зміненої позиції.

**Крайовий випадок — деактивація регіону:** виключені регіони пропускаються при визначенні черги без порушення дати для наступного активного регіону; вже згенеровані минулі патерни для деактивованого регіону лишаються доступними в архіві незмінними.

## 8.4. Математична основа симетрії

Побудова орнаменту спирається на формальний математичний апарат, яким описується традиційна вишивка: групи відбиття та групи шпалерного типу. Обраний мотив компонується за правилами конкретної групи симетрії, властивої регіону чи типу орнаменту.

## 8.5. Накладання на силует сорочки

Готовий орнамент накладається на заздалегідь підготовлений контур сорочки (комір, планка, манжети) у зонах, що відповідають реальним місцям вишивки на традиційному одязі. Контур сорочки — статичний елемент дизайну.

## 8.6. Стійкість до збоїв генерації

Оскільки лінива генерація відбувається синхронно в межах живого HTTP-запиту (розділ 11), збій алгоритму не повинен призводити до повної відмови сторінки для першого відвідувача дня. Передбачений резервний сценарій: якщо виклик сервісу генерації завершується винятком — система записує `DailyPattern` зі статусом «помилка, застосовано резервний варіант» і показує востаннє успішно згенерований патерн (учорашній) з чітким текстовим поясненням («сьогоднішній орнамент тимчасово недоступний, ось учорашній»), а не порожню сторінку чи технічну помилку. Подія збою обов'язково логується для подальшої ручної діагностики персоналом.

## 8.7. Формат результату

Кінцевий результат — текстовий вміст у форматі масштабованої векторної графіки. Не вимагає растрового рендерингу на сервері, важить незначно, відмінно масштабується без втрати якості, легко зберігається як текстовий стовпець бази даних.

## 8.8. Версіонування алгоритму

Кожен згенерований запис фіксує номер версії алгоритму. Історичні записи залишаються незмінними при зміні логіки генерації в майбутньому.

# 9. Колекціонування та особистий прогрес

## 9.1. Логіка збереження

Проста операція додавання/видалення зв'язку між користувачем і конкретним патерном. Обмеження унікальності на рівні бази даних запобігає дублюванню.

## 9.2. Індикатор «пройденого туру»

Обчислюється на льоту при відображенні сторінки колекції: кількість унікальних регіонів, представлених у збережених патернах, віднесена до загальної кількості активних регіонів. Не є окремою збереженою сутністю.

## 9.3. Права доступу

Колекція — виключно приватна, немає публічного перегляду чужих колекцій.

# 11. Фонові процеси і «лінива генерація» (заміна класичного планувальника)

## 11.1. Чому традиційний планувальник не підходить

На безкоштовному тарифі цільового хостингу відсутня можливість запускати задачі за розкладом. Традиційний підхід «щоночі генерувати контент через фоновий планувальник» — технічно неможливий без переходу на платний тариф.

## 11.2. Механізм лінивої генерації

Перевірка відбувається при кожному запиті до головної сторінки: контролер перевіряє наявність запису для сьогоднішньої дати. Якщо запису немає — синхронно викликається сервіс генерації, результат зберігається, і той самий запит одразу отримує щойно згенерований контент.

**Наслідок для продуктивності:** генерація відбувається щонайбільше один раз на добу, триває дуже коротко, тому додаткове навантаження на обмежений добовий ліміт обчислювального часу хостингу — мінімальне.

**Захист від одночасного подвійного запуску:** обмеження унікальності на рівні бази даних на полі дати гарантує, що навіть за одночасних запитів буде створено лише один запис.

## 11.3. Що робити з майбутніми фоновими задачами

Функціонал, що потребує повторюваних дій без прив'язки до конкретного запиту (майбутній щотижневий дайджест, сповіщення про заявку на гостьовий пост) — на етапі безкоштовного хостингу реалізується синхронно в момент дії користувача, або відкладається до переходу на платний тариф.

## 11.4. Шлях масштабування

При переході на платний тариф — функціонал ліниво генерованого контенту не потребує змін, проте з'являється можливість додатково підстрахуватися завчасною генерацією наступного дня через реальний плановий процес.

## 13.2. Захист від міжсайтової підробки запиту

Стандартний вбудований механізм для всіх форм; для асинхронних дій — токен через заголовок запиту.

## 13.3. Захист від міжсайтового виконання скриптів

Автоматичне екранування шаблонізатора застосовується всюди за замовчуванням. Тіло статей блогу — санітизація на сервері при збереженні з чітким переліком дозволених тегів і атрибутів.

## 13.4. Захист від SQL-ін'єкцій

Об'єктно-реляційне відображення параметризує запити за замовчуванням.

## 13.5. Захист від перебору паролів

Механізм блокування за кількістю невдалих спроб входу.

## 13.6. Політика безпеки вмісту

Обмеження джерел завантаження скриптів, стилів і зображень виключно власним доменом, заборона вбудовування сторінки в чужі фрейми.

## 13.7. Транспортна безпека

Примусове перенаправлення на захищене з'єднання, тривалий термін дії суворого транспортного захисту, захищені атрибути кук.

## 13.8. Валідація посилань на зображення

Стосується виключно посилання на обкладинку статті блогу. Перелік дозволених доменів, перевірка схеми з'єднання, обережність щодо запитів на внутрішні мережеві адреси.

## 13.9. Керування секретами

Усі чутливі значення зберігаються поза кодовою базою, у змінних середовища (детально розділ 19.5), з прикладом-шаблоном без реальних значень у контролі версій.

## 13.10. Захист адміністративної панелі

Нетривіальна, налаштовувана адреса адміністративної панелі. Суворо заданий перелік дозволених доменів. Сильні паролі персоналу, вимкнений режим налагодження на продакшн-середовищі за жодних обставин.

## 13.11. Захист форм від зловживання третіми особами

Прихована honeypot-перевірка на контактній формі й формі заявки на гостьовий пост.

## 13.12. Конфіденційність даних користувачів

Мінімальний обсяг персональних даних, що збираються (електронна адреса, необов'язкове відображуване ім'я) — свідомо мінімалістична модель профілю знижує ризики й спрощує відповідність вимогам конфіденційності. Контактні повідомлення й заявки на гостьові пости зберігають IP-адресу відправника виключно для боротьби зі зловживаннями — не використовуються для жодної іншої мети. Політика конфіденційності (стаття статичної сторінки) явно описує який мінімальний обсяг даних збирається і з якою метою.

# 14. Адміністративна панель (Django Unfold + CKEditor 5)

## 14.1. Загальна конфігурація

Кастомна тема адміністративної панелі, з налаштуванням: назва і заголовок панелі, іконка/логотип бренду, кольорова палітра (базується на кольорах вишивки — червоний і чорний як акценти на нейтральному тлі), згруповане бічне меню.

## 14.2. Групування бічного меню

- **«Контент дня»** — щоденні патерни (переважно перегляд, оскільки генерація автоматична).
- **«Довідники»** — регіони, мотиви, джерела (основна ручна робота адміністратора при наповненні продукту).
- **«Блог»** — статті, категорії статей, заявки на гостьові пости.
- **«Спільнота»** — користувачі, профілі, збережені патерни.
- **«Звернення»** — контактні повідомлення.
- **«Сторінки»** — статичні сторінки, пункти FAQ.

## 14.3. Дашборд із віджетами

- Лічильник загальної кількості зареєстрованих користувачів і приріст за останній тиждень.
- Лічильник нових заявок на гостьові пости, що очікують розгляду.
- Лічильник непрочитаних контактних звернень.
- Візуалізація розподілу збережених патернів за регіонами.
- **Лічильник регіонів/мотивів зі статусом «на перевірці» чи «потребує перегляду»** — практичний індикатор редакційного боргу, прямо пов'язаний із методологією достовірності (розділ 4).
- **Лічильник патернів зі статусом «помилка, застосовано резервний варіант»** — індикатор технічних збоїв генерації, що потребують уваги.
- Останні кілька опублікованих статей блогу з швидким переходом до редагування.

## 14.4. Реєстрація моделей

**Регіон:** список відображає назву, кількість пов'язаних мотивів, статус активності, статус верифікації, порядок ротації; пошук за назвою; масові дії активування/деактивації; редаговані переклади через вкладки мов; вбудований перелік пов'язаних джерел прямо на сторінці редагування регіону.

**Мотив:** аналогічно, з фільтрацією за сумісними регіонами й статусом верифікації.

**Джерело:** список із типом, автором/установою, кількістю пов'язаних регіонів/мотивів; пошук за назвою й автором.

**Щоденний патерн:** переважно перегляд, фільтрований за датою, регіоном, статусом генерації; поля результату генерації — лише для читання; окрема дія «примусово перегенерувати» для виняткових випадків з явним попередженням про створення нового результату поточною версією алгоритму замість збереженого.

**Стаття блогу:** повний редактор WYSIWYG для тіла статті; поля типу публікації з обов'язковим заповненням імені партнера при непередакційному типі; попереднє автоматичне заповнення slug; вбудований перелік пов'язаних джерел; список відображає статус, тип публікації, категорію, дату публікації, лічильник переглядів; фільтрація за статусом і типом; переклад полів через вкладки мов.

**Заявка на гостьовий пост:** переважно для читання; редагується лише статус розгляду; масове позначення розглянутим.

**Контактне повідомлення:** для читання, з дією позначення прочитаним.

## 14.5. Конфігурація редактора статей

Панель інструментів включає базове форматування (заголовки, жирний/курсив текст, посилання, списки, цитати, завантаження зображень, таблиці, вбудований код, режим вихідного коду). Дозволений перелік HTML-тегів і атрибутів обмежений навмисно. Завантаження зображень у статтях обмежене за розміром і типом файлу, доступне лише персоналу.

Мінімізація стилів і скриптів, самостійний хостинг шрифтів, відсутність зайвих растрових зображень (основний контент — векторна графіка), уникнення зсуву макета через фіксовані пропорції контейнера силуету сорочки незалежно від стану завантаження.

# 16. Мультимовність

## 16.1. Вибір основної мови

Українська — основна мова, без префіксу в адресі. Англійська — другорядна, з префіксом.

## 16.2. Структура адрес

Загальні маршрути обгортаються мультимовною логікою маршрутизації з явним налаштуванням, що основна мова не отримує префікса. Службові розділи — поза мультимовною обгорткою.

## 16.3. Переклад інтерфейсу

Позначення рядків для перекладу в коді й шаблонах, витяг рядків в окремі файли перекладу, ручний переклад, компіляція перед розгортанням.

## 16.4. Переклад контенту моделей

Реєстрація перекладних полів для регіону, мотиву, категорій і статей блогу, статичних сторінок, пунктів FAQ. Фолбек на основну мову за відсутності перекладу.

## 16.5. Робочий процес перекладу культурного контенту — окреме правило

На відміну від технічних інтерфейсних рядків (де машинний переклад прийнятний як початкова точка з подальшою вичиткою), **описи символіки регіонів і мотивів пишуться спочатку українською як основний, авторитетний варіант, а англійська версія створюється через свідомий людський переклад із перевіркою культурної й термінологічної точності, не через автоматичний машинний переклад без подальшої редактури.** Помилковий чи неточний переклад культурного значення символу — репутаційний ризик для проєкту, що прямо суперечить меті достовірності (розділ 4). Це правило фіксується як обов'язкова частина редакційного робочого процесу, не як побажання.

## 16.6. Що ніколи не перекладається

Службові й технічні поля щоденного патерну, особисті дані користувачів, вміст контактних звернень і заявок на гостьові пости, бібліографічні дані джерел.

## 16.7. Перемикач мови

Невелика форма в підвалі сайту, що передає намір зміни мови на службовий обробник, який зберігає вибір у кукі й сесії та перенаправляє на той самий контент іншою мовою.

## 16.8. SEO для мультимовного контенту

Альтернативні мовні посилання в заголовку кожної сторінки, окремі записи в карті сайту з альтернативами, канонічна адреса завжди вказує на версію поточної активної мови.

# 17. Бекап та відновлення

## 17.1. Фундаментальний принцип, специфічний для цього продукту

На відміну від типової платформи з даними, що безповоротно втрачаються при пошкодженні бази, **основний обсяг генерованого контенту цього продукту — детерміновано відтворюваний**. Якщо запис щоденного патерну буде втрачено, він може бути перегенерований точно в такому самому вигляді за умови збереження вхідних даних для генерації: дати, ідентифікатора регіону, seed-значення і версії алгоритму.

## 17.2. Класифікація даних за критичністю

| Категорія даних                                  | Критичність втрати | Обґрунтування                                                                                                                                                        |
| ------------------------------------------------ | ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Щоденні патерни (сам SVG-результат)              | Низька             | Відтворюваний детерміновано                                                                                                                                          |
| Регіони, мотиви і **джерела**                    | **Найвища**        | Ручна авторська й дослідницька робота з наповнення й верифікації символіки — не відтворюється автоматично; втрата означає повторення всієї редакційної роботи з нуля |
| Статті блогу, особливо гостьові/спонсоровані     | **Найвища**        | Унікальний контент, часто наданий третьою стороною за домовленістю                                                                                                   |
| Облікові записи користувачів і профілі           | Висока             | Втрата означає втрату аудиторії, що реєструвалась                                                                                                                    |
| Збережені колекції користувачів                  | Середня            | Незручність при втраті, але не критична для бізнесу                                                                                                                  |
| Заявки на гостьові пости, контактні повідомлення | Середня            | Історичний слід ділового спілкування                                                                                                                                 |

## 17.3. Стратегія бекапу на етапі безкоштовного хостингу

- Періодичне (не рідше разу на тиждень на етапі активного наповнення довідників, не рідше разу на місяць у сталому режимі) ручне експортування вмісту бази даних у портативний формат через консоль керування хостингу.
- Отриманий файл вивантажується в окреме, незалежне від основного хостингу сховище.
- **Довідники регіонів, мотивів і джерел, як найкритичніший вручну створений і верифікований контент, додатково дублюються в текстовому, людинозрозумілому форматі окремо від загального знімка бази** — дозволяє відновити символьні описи й посилання на джерела навіть без повного відновлення бази даних.

## 17.4. Стратегія бекапу після переходу на платний план чи власний сервер

Автоматизоване регулярне резервне копіювання засобами системи керування базами даних, за розкладом, зі зберіганням кількох останніх версій і вивантаженням в окреме зовнішнє сховище об'єктів.

## 17.5. Процедура відновлення — концептуальний опис

Спочатку відновлюється найкритичніша категорія (довідники регіонів, мотивів, джерел, статті блогу, облікові записи) з останнього знімка. Щоденні патерни за період після останнього знімка можуть бути перегенеровані повторним запуском сервісу генерації для кожної відсутньої дати за умови, що версія алгоритму відповідає версії, зафіксованій у втраченому записі.

## 17.6. Тестування процедури відновлення

Періодична практична перевірка, що наявний знімок бази даних дійсно піддається коректному розгортанню в чистому тестовому середовищі.

# 18. Деплой

## 18.1. Фаза 1 — безкоштовний тариф хостингу

**Обмеження середовища, з якими свідомо узгоджена архітектура:**

- Відсутність можливості запуску задач за розкладом — компенсовано механізмом лінивої генерації.
- Обмежений добовий ліміт обчислювального часу — прийнятно, оскільки генерація легка й одноразова на день.
- Обмежений дисковий простір — прийнятно, жодні зображення не зберігаються як файли.
- База даних — вбудований файловий різновид, без потреби в окремому сервері бази даних.
- Обмежений список дозволених зовнішніх адрес — не є проблемою, оскільки жодна частина функціоналу не звертається до зовнішніх програмних інтерфейсів.

**Загальна послідовність розгортання:** створення облікового запису на хостингу, отримання коду проєкту на сервер, налаштування ізольованого середовища виконання Python зі встановленням залежностей, налаштування змінних середовища, конфігурація точки входу веб-застосунку з посиланням на продакшн-конфігурацію налаштувань, застосування міграцій бази даних, створення першого адміністративного облікового запису, збирання статичних файлів, компіляція перекладів, **завантаження початкового набору регіонів, мотивів і джерел через підготовлені фікстури чи керовану команду** (детально розділ 18.5), перезапуск веб-застосунку.

**Оновлення коду після першого розгортання:** отримання оновленого коду, за потреби — оновлення залежностей, застосування нових міграцій, повторне збирання статичних файлів, перезапуск застосунку.

## 18.2. Необхідні змінні середовища (концептуальний перелік)

Секретний ключ підпису застосунку; перемикач режиму налагодження (обов'язково вимкнений на продакшн); перелік дозволених доменів застосунку; рядок підключення до бази даних (порожній чи файловий шлях для вбудованого варіанту, повний рядок підключення для зовнішнього керованого сервера при масштабуванні); налаштування відправлення пошти (спосіб відправлення — консольний вивід для локальної розробки, справжній поштовий сервер для продакшн; облікові дані поштового сервера); ідентифікатор і секрет клієнта для входу через Google; нетривіальна адреса адміністративної панелі; за потреби — ключ підключення до сервісу моніторингу помилок (розділ 18.6). Усі значення — поза кодовою базою, приклад-шаблон без реальних значень зберігається в контролі версій, реальний файл зі значеннями — явно виключений.

## 18.3. Необхідні залежності (концептуальний перелік, без версій)

Основний фреймворк; адаптер бібліотеки автентифікації з підтримкою входу через Google; кастомна тема адміністративної панелі; редактор WYSIWYG для блогу; бібліотека перекладу моделей для мультимовного контенту; бібліотека транслітерації для генерації slug; бібліотека читання змінних середовища; бібліотека санітизації HTML для тіла статей блогу; бібліотека обмеження частоти запитів; бібліотека захисту від перебору паролів; окремо для локальної розробки — інструмент якості коду й форматування, тестовий фреймворк, бібліотека генерації тестових об'єктів; окремо для продакшн — WSGI-сервер (при переході на власний сервер), бібліотека роздачі статики (за потреби), клієнт сервісу моніторингу помилок.

## 18.4. Порогові індикатори для переходу на вищий рівень

Систематичне наближення до добового ліміту обчислювального часу через зростання трафіку; потреба у справжньому плановому процесі; потреба у власному домені верхнього рівня; потреба в керованій реляційній базі даних через зростання обсягу паралельних запитів на запис.

## 18.5. Чек-лист перед першим публічним запуском

Обов'язкові умови, перевірені перед відкриттям продакшн-сайту для реальних відвідувачів:

- Мінімальний життєздатний набір регіонів наповнено й переведено в статус «верифіковано» (рекомендований мінімум — достатня кількість для повного календарного місяця без надто швидкого повторення одного й того самого регіону; конкретне число уточнюється на етапі наповнення, але свідомо не менше кількох десятків днів унікальної ротації).
- Кожен активний регіон і мотив має щонайменше одне пов'язане джерело.
- Резервний сценарій на випадок збою генерації (розділ 8.6) протестований — штучно викликана помилка генерації дійсно показує вчорашній патерн, а не технічну помилку.
- Політика конфіденційності й умови використання опубліковані як статичні сторінки.
- Режим налагодження вимкнений, перевірені сторінки 404 і 500 у продакшн-режимі.
- Перевірене відправлення листів (підтвердження реєстрації, відновлення паролю) працює коректно на реальному поштовому налаштуванні, не лише в консольному режимі локальної розробки.

## 18.6. Моніторинг помилок

Підключення стороннього сервісу моніторингу помилок для застосунку — фіксує неопрацьовані винятки в продакшн-режимі до того, як користувач побачить загальну сторінку помилки, з деталізованим трасуванням для розробника. Особливо важливо для сервісу генерації орнаменту (розділ 8.6) — кожен випадок застосування резервного сценарію має бути видимим розробнику через сервіс моніторингу, не лише мовчки записаним у службовий статус бази даних.

## 18.7. Проміжний крок — платний тариф того самого хостингу

Найменш витратний за зусиллями наступний крок: перехід на платний рівень тієї самої платформи хостингу додає можливість планових задач, знімає обмеження дискового простору й обчислювального часу, дозволяє підключення повноцінної керованої бази даних без зміни коду.

## 18.8. Фаза 2 — власний сервер (за потреби значного масштабування)

Перехід на власний віртуальний сервер за типовою логікою продакшн Django-проєкту: сервер застосунків за реверс-проксі, окремий процес для планових задач (якщо на цьому етапі вже впроваджено функціонал, що цього потребує), окрема повноцінна керована база даних, автоматичне поновлення сертифіката захищеного з'єднання, налаштований міжмережевий екран, автоматичні оновлення безпеки операційної системи, моніторинг стану всіх запущених процесів.

**Ключова перевага свідомо простої архітектури цього проєкту:** оскільки немає залежності від довгоживучих фонових воркерів на першому етапі, перехід на власний сервер не вимагає негайного розгортання складної інфраструктури черг повідомлень — вона додається лише тоді, коли з'являється реальний функціонал, що її потребує.

# 19. Контейнеризація для локальної розробки

## 19.1. Призначення

Хоча продакшн-середовище на першому етапі не використовує контейнеризацію, локальна розробка виграє від контейнеризованого підходу для узгодженості середовища між різними машинами розробки й майбутньою легкістю переходу на власний сервер.

## 19.2. Необхідні сервіси для локального середовища

Основний сервіс застосунку Django. За потреби (якщо на етапі локальної розробки використовується повноцінна реляційна база даних замість файлового різновика для точнішого відтворення майбутнього продакшн-середовища) — окремий сервіс бази даних. Оскільки жодна фонова черга задач не потрібна на першому етапі — сервіси кешу в оперативній пам'яті чи брокера повідомлень не є обов'язковими для локального середовища цього конкретного проєкту.

## 19.3. Загальний підхід до організації

Окремий опис середовища виконання (базовий образ, системні залежності, встановлення залежностей Python) та окремий файл оркестрації сервісів для локального запуску одною командою, з монтуванням коду проєкту як тому для миттєвого відображення змін без перезбирання образу під час активної розробки.

# 20. Безперервна інтеграція та якість коду

## 20.1. Загальний підхід

Автоматизована перевірка якості коду й запуск тестів при кожній зміні в основній гілці репозиторію та при кожній пропозиції змін.

## 20.2. Етапи автоматизованої перевірки

Перевірка стилю й потенційних помилок коду єдиним сучасним інструментом. Запуск повного набору автоматизованих тестів із формуванням звіту покриття. За успішного проходження — розгортання оновленого коду на цільовому середовищі.
Розділ 20.2, уточнення: Єдиний інструмент перевірки стилю й форматування — Ruff, той самий що використовувався в SetupHunt. Підключається як пре-коміт хук з першого дня розробки, до першого коміту з кодом моделей.

## 20.3. Локальна перевірка перед відправленням змін

Той самий інструмент якості коду підключається як локальний перед-комітний гачок.

## 20.4. Секрети для автоматизованого розгортання

За потреби автоматизованого розгортання на власний сервер — реквізити доступу до сервера зберігаються в захищеному сховищі секретів системи керування версіями.

# 21. Тестування

## 21.1. Що потребує обов'язкового покриття тестами

- **Детермінованість генерації** — найкритичніший тестовий сценарій усього проєкту: повторний виклик сервісу генерації з тим самим початковим значенням і тією самою версією алгоритму повинен завжди повертати ідентичний результат.
- **Резервний сценарій при збої генерації** — штучно змодельована помилка алгоритму дійсно приводить до показу вчорашнього патерна, а не необробленого винятку.
- **Логіка ротації регіонів** — коректність визначення, який регіон відповідає якій даті, включно з коректною обробкою тимчасово вимкнених і щойно доданих регіонів.
- **Ідемпотентність лінивої генерації** — повторний запит того самого дня не створює другого запису патерну.
- **Обмеження унікальності** — неможливість дублювання збереженого патерну для того самого користувача, неможливість двох патернів на ту саму дату.
- **Обов'язковість джерела** — неможливість перевести регіон чи мотив у статус «верифіковано» без пов'язаного джерела.
- **Права доступу** — недоступність чужої колекції, недоступність чернеток статей блогу для звичайних відвідувачів, недоступність адміністративних дій для неавторизованого персоналу.
- **Форми та захист від спаму** — коректна поведінка honeypot-перевірки, обмеження частоти подання.
- **Мультимовність** — коректне відображення перекладних полів залежно від активної мови, коректна робота фолбеку за відсутності перекладу.

## 21.2. Підхід до підготовки тестових даних

Фабрики об'єктів для гнучкого породження тестових сутностей (користувач, регіон, мотив, джерело, щоденний патерн) із розумними значеннями за замовчуванням. Статичні файли початкових даних — лише для справді довідкового вмісту.

## 21.3. Ціль покриття

Орієнтовна ціль — не менше вісімдесяти відсотків покриття кодовою базою, з обов'язковим стовідсотковим покриттям для сервісу генерації орнаменту окремо, з огляду на його критичність для коректності всього продукту.
