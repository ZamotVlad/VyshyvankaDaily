
# VyshyvankaDaily — Roadmap

> Покроковий план розробки від порожнього репозиторію до продакшн-запуску.
> Кожна стадія має мету, критерій готовності («Definition of Done»), перелік задач і, де доречно — застереження щодо ризиків.
> Посилання на розділи `TZ_VYSHYVANKA.md` наведені для трасованості між планом і специфікацією.

---

## Принципи цього плану

**Порядок стадій визначений ризиком, не зручністю.** Найризикованіша й найменш перевірена частина проєкту — алгоритм генерації орнаменту — йде одразу після фундаменту, до того, як будується весь інтерфейс навколо неї.

**Дослідницька робота (Stage 5) свідомо відокремлена від програмування.** Пошук і перевірка достовірних джерел символіки — це не рядки коду, і оцінити її тривалість наперед складніше, ніж оцінити написання view.

**Тести пишуться разом із кодом, не в кінці.** Особливо для сервісу генерації — розділ 21.1 ТЗ вимагає стовідсоткового покриття саме цього модуля.

**Ruff підключається до першого коміту з кодом**, не після.

**Одна гілка — одна задача — один змерджений результат.**

**Реалістичний темп:** оцінки нижче розраховані на соло-розробника, що працює вечорами після основної зайнятості. Якщо якась стадія йде важче за прогноз — переносити наступну, не стискати її.

## Stage 0 — Фундамент, інфраструктура і якість коду

**Мета:** середовище готове настільки, що з наступного дня можна писати лише прикладний код, не витрачаючи час на конфігурацію.

**Орієнтовна тривалість:** ~1 тиждень.

### Задачі

- [ ] Ініціалізація Git-репозиторію, `.gitignore` (включно з винятками для середовища виконання, зібраної статики, файлу бази даних локальної розробки)
- [ ] Приклад файлу змінних середовища (без реальних значень) — шаблон за розділом 18.2 ТЗ
- [ ] **Ruff** — конфігурація лінтера й форматера, підключення як pre-commit хук, перший прогін на порожньому/мінімальному проєкті, щоб зафіксувати чистий старт
- [ ] **Контейнеризація локальної розробки** (розділ 19 ТЗ): опис середовища виконання й файл оркестрації сервісів для локального запуску одною командою, з монтуванням коду як тому для миттєвого відображення змін
- [ ] **Базовий конвеєр безперервної інтеграції** (розділ 20 ТЗ): автоматична перевірка лінтером і запуск тестів при кожному push і pull request — навіть якщо тестів на цьому етапі ще мало, конвеєр налаштовується одразу, щоб зростати разом із проєктом, а не підключатися заднім числом
- [ ] Структура проєкту: конфігураційний пакет (розбиті за середовищем налаштування — базові/локальні/тестові/продакшн), директорія застосунків
- [ ] Створення застосунків: `core`, `accounts`, `patterns`, `blog`, `pages`
- [ ] Абстрактні моделі в `core`: модель часових позначок, модель ЧПУ-ідентифікатора з логікою транслітерації (розділ 3.1 ТЗ)
- [ ] Підключення `django-modeltranslation` **з нульового дня**, до першої міграції моделей, що потребують перекладу
- [ ] Базові налаштування середовищ, читання змінних середовища через відповідну бібліотеку
- [ ] Перевірка, що застосунок запускається локально (у контейнері) без помилок конфігурації

### Критерій готовності (Definition of Done)

Порожній Django-проєкт із правильною структурою застосунків, підключеним Ruff, працюючим локальним контейнерним середовищем, базовим CI-конвеєром, що зелено проходить на порожньому наборі перевірок, і мультимовною інфраструктурою — запускається однією командою без помилок, перший коміт проходить pre-commit хук чисто.

## Stage 1 — Ядро генерації орнаменту (найризикованіший блок, пріоритет)

**Мета:** довести технічну реалізовність центральної унікальної механіки продукту, перш ніж інвестувати час у побудову інтерфейсу навколо неї.

**Орієнтовна тривалість:** ~2–3 тижні. Це стадія з найбільшою невизначеністю тривалості в усій кодовій частині плану.

### Задачі

- [ ] Моделі `Region`, `Motif`, `Source` з полями за розділами 3.2–3.4 ТЗ, включно з полем статусу верифікації й обов'язковим зв'язком із джерелом
- [ ] Модель `DailyPattern` з полями seed, версії алгоритму, статусу генерації (розділ 3.5 ТЗ)
- [ ] Модель `SavedPattern` (можна створити зараз навіть без інтерфейсу колекції — знадобиться в Stage 3)
- [ ] Перша міграція, реєстрація перекладних полів для `Region` і `Motif`
- [ ] Сервісний шар генерації, відокремлений від будь-якого HTTP-контексту
- [ ] Реалізація перетворення дати на детермінований seed
- [ ] Реалізація базової геометрії мотиву для щонайменше однієї групи відбиття й однієї групи шпалерного типу
- [ ] Побудова контуру силуету сорочки й логіка накладання орнаменту на визначені зони
- [ ] Логіка ротації регіонів, включно з коректною обробкою деактивованих регіонів (розділ 8.3 ТЗ)
- [ ] Резервний сценарій при винятку в процесі генерації (розділ 8.6 ТЗ)
- [ ] **Тести детермінованості** — найкритичніший тест усього проєкту
- [ ] Тести ротації регіонів і резервного сценарію
- [ ] **Проміжна контрольна точка (емоційний чекпоінт):** одна тимчасова debug-сторінка (не повноцінний публічний view, лише для власної перевірки), що рендерить результат сервісу генерації прямо в браузері для довільної тестової дати — мета не архітектурна, а мотиваційна: побачити перший живий орнамент на екрані ще до завершення всієї стадії, не чекаючи Stage 2 повністю

### Критерій готовності

Виклик сервісу генерації в консолі розробки для довільної дати повертає коректний SVG-текст, повторний виклик тим самим параметром дає ідентичний результат, тести детермінованості проходять стабільно, резервний сценарій підтверджено спрацьовує, і — суб'єктивно, але важливо — ти вже бачив орнамент на власні очі в браузері хоча б раз до завершення стадії.

### Застереження

Якщо повна математична реалізація груп симетрії потребує значно більше часу за заплановане — прийнятний компроміс: спростити геометричні правила для першої версії, зафіксувати це рішення коротким записом, розширювати різноманітність мотивів ітеративно вже після запуску.

## Stage 2 — Публічні сторінки MVP

**Мета:** продукт відкривається в браузері, і в ньому реально видно щоденний орнамент.

**Орієнтовна тривалість:** ~2 тижні.

**Залежність:** потребує завершеного Stage 1.

### Задачі

- [ ] Головна сторінка: логіка лінивої генерації (перевірка існування запису на сьогодні → виклик сервісу генерації за відсутності → рендер) — розділ 11.2 ТЗ
- [ ] Базовий шаблон, шапка, підвал із логікою видимості без JavaScript
- [ ] Сторінка конкретного патерну за датою (`/pattern/<дата>/`)
- [ ] Сторінка регіону з описом символіки й блоком джерел
- [ ] Архів із фільтрами за регіоном і діапазоном дат, пагінація
- [ ] Base-стилі Bootstrap 5, адаптація силуету сорочки під різні розміри екрана
- [ ] **Сторінки помилок 404 і 500** (розділ 5.14 ТЗ): дружні, мінімалістичні шаблони; окрема увага до 500-сторінки — мінімум логіки, без запитів до бази даних і без context processors, що звертаються до бази, оскільки саме база може бути причиною помилки
- [ ] Мінімальний набір тестових/тимчасових регіонів і мотивів для локальної розробки (не фінальний достовірний контент)
- [ ] Тести: статус-коди основних сторінок, коректна поведінка при запиті неіснуючої дати чи неактивного регіону, коректний виклик сторінки 404 для неіснуючих адрес

### Критерій готовності

Локальний запуск проєкту показує на головній сторінці силует сорочки зі згенерованим орнаментом дня, перехід в архів і на сторінку окремого регіону працює коректно, підвал поводиться згідно з описаною логікою, сторінки 404/500 показуються коректно й не падають повторно навіть при штучно відключеній базі даних.

## Stage 3 — Авторизація і колекція

**Мета:** з'являється персональний шар — реєстрація, вхід, можливість зберігати улюблені патерни.

**Орієнтовна тривалість:** ~1–2 тижні.

### Задачі

- [ ] **Модель `Profile`** (розділ 3.7 ТЗ) — зв'язок «один-до-одного» з користувачем, поле відображуваного імені, поле мови інтерфейсу за замовчуванням; міграція
- [ ] Підключення бібліотеки автентифікації: вхід за електронною поштою, обов'язкова верифікація пошти, вхід через Google (розділ 7 ТЗ)
- [ ] Сигнал автоматичного створення `Profile` при реєстрації, з перевіркою прапорця «сирого» збереження (щоб не зламати завантаження тестових фікстур)
- [ ] Сторінка налаштувань профілю
- [ ] Логіка збереження/видалення патерну в колекцію (`SavedPattern`), з обмеженням унікальності на рівні бази
- [ ] Сторінка «Моя колекція» з переліком збережених патернів
- [ ] Індикатор «пройденого туру» — обчислення на льоту частки унікальних регіонів у колекції
- [ ] Тести прав доступу: анонім не бачить кнопку збереження, чужа колекція недоступна, власник керує лише своїм профілем

### Критерій готовності

Реєстрація, вхід через пошту й через Google, збереження патерну в колекцію та його відображення на сторінці колекції — усе працює наскрізно вручну перевірено, `Profile` коректно створюється автоматично для кожного нового користувача.

## Stage 4 — Блог і повноцінна адміністративна панель

**Мета:** з'являється канал для SEO-контенту й монетизаційної моделі через партнерські матеріали, а також зручний інструмент керування всім довідниковим контентом.

**Орієнтовна тривалість:** ~2 тижні.

### Задачі

- [ ] Моделі `BlogCategory`, `BlogPost` (з полем типу публікації — редакційна/гостьова/спонсорована) і `GuestPostSubmission`
- [ ] Сторінки блогу: список, окрема стаття, з коректним розрізненням статусів і позначкою «Партнерський матеріал» для непередакційного типу
- [ ] Форма заявки на гостьовий пост, з honeypot-перевіркою й обмеженням частоти подання
- [ ] Підключення кастомної теми адміністративної панелі (Django Unfold)
- [ ] Групування бічного меню за розділом 14.2 ТЗ
- [ ] **Дашборд із віджетами** (розділ 14.3 ТЗ): лічильник користувачів і приросту, лічильник заявок на гостьові пости, лічильник непрочитаних звернень, візуалізація розподілу збережень за регіонами, лічильник записів «на перевірці»/«потребує перегляду», лічильник патернів «помилка генерації»
- [ ] Реєстрація всіх моделей в адміністративній панелі згідно з розділом 14.4 ТЗ
- [ ] Підключення редактора CKEditor 5 для тіла статей, з обмеженим переліком дозволених HTML-тегів і санітизацією при збереженні
- [ ] Статичні сторінки, FAQ, контактна форма (застосунок `pages`)
- [ ] Тести: чернетки статей недоступні звичайним відвідувачам, заявка на гостьовий пост коректно обробляє honeypot і обмеження частоти

### Критерій готовності

Адміністративна панель відкривається кастомною темою, дашборд показує реальні цифри, можна створити й опублікувати статтю блогу через WYSIWYG-редактор, можна переглянути й обробити тестову заявку на гостьовий пост.

## Stage 5 — Дослідження і наповнення контентом

**Мета:** мінімальний життєздатний набір регіонів наповнений достовірним, перевіреним контентом — умова публічного запуску.

**Орієнтовна тривалість:** свідомо не оцінюється в тижнях програмної роботи. Планувати паралельно з попередніми стадіями.

### Задачі

- [ ] Складання переліку історичних регіонів України для першої версії ротації
- [ ] Пошук джерел за ієрархією довіри з розділу 4.2 ТЗ
- [ ] Заповнення моделі `Source` для кожного знайденого джерела
- [ ] Написання описів символіки для кожного регіону українською, з розрізненням задокументованого й традиційного
- [ ] Наповнення довідника мотивів із перевіркою на неоднозначні сучасні конотації
- [ ] Переведення кожного регіону й мотиву зі статусу «на перевірці» в «верифіковано» лише після підтвердженого джерела
- [ ] Людський переклад описів символіки на англійську
- [ ] Написання перших кількох статей блогу на основі вже дослідженого матеріалу

### Критерій готовності

Кількість верифікованих регіонів достатня для повного календарного місяця ротації. Кожен активний регіон і мотив має щонайменше одне джерело. Усі описи мають чітке маркування типу твердження.

### Застереження

Розумний компроміс: для запуску достатньо кількості регіонів, що дає місяць унікальної ротації з дійсно перевіреним контентом — решту можна додавати поступово вже після запуску.

## Stage 6 — Якість, безпека, SEO, доступність, продуктивність

**Мета:** продукт готовий витримати реальний, непередбачуваний інтернет-трафік і коректно індексуватися пошуковими системами.

**Орієнтовна тривалість:** ~1–2 тижні.

### Задачі

- [ ] Обмеження частоти запитів на всі чутливі точки — розділ 13.1 ТЗ
- [ ] Захист від перебору паролів на вході
- [ ] Політика безпеки вмісту, налаштування транспортної безпеки
- [ ] Валідація посилань на обкладинки статей блогу
- [ ] Нетривіальна адреса адміністративної панелі
- [ ] **Стратегія кешування** (розділ 12 ТЗ): свідоме рішення — на етапі безкоштовного хостингу додатковий шар кешування в оперативній пам'яті не впроваджується (природний кеш — сама база даних, оскільки контент дня детермінований і не змінюється), рішення задокументоване; підготовлено чіткий список кандидатів на кешування (фрагмент головної сторінки, список блогу, статичні сторінки) для впровадження при зростанні трафіку, без потреби переробляти архітектуру заднім числом
- [ ] Meta-теги для кожного типу сторінки, Open Graph і картки для соціальних мереж
- [ ] Карта сайту з мультимовними альтернативами, файл інструкцій для пошукових роботів
- [ ] Канонічні адреси для сторінок з фільтрами
- [ ] Структуровані дані для статей блогу й сторінок патерну
- [ ] Перевірка ієрархії заголовків на кожному типі сторінки
- [ ] Доступність: текстовий описовий атрибут для кожного SVG-патерна, перевірка контрастності інтерфейсних елементів
- [ ] **Продуктивність відображення** (розділ 15.9 ТЗ): мінімізація стилів і скриптів, самостійний хостинг шрифтів (без залежності від зовнішніх джерел), фіксовані пропорції контейнера силуету сорочки для уникнення зсуву макета під час завантаження
- [ ] Повний прогін мультимовності: усі інтерфейсні рядки позначені для перекладу, переклади зібрані й скомпільовані, перевірка перемикача мови й фолбеку
- [ ] Політика конфіденційності й умови використання як статичні сторінки
- [ ] Розширення набору автоматизованих тестів до цільового покриття (не менше 80% загалом, 100% для сервісу генерації)

### Критерій готовності

Повний прогін продукту обома мовами без помилок перекладу, перевірка основних сторінок інструментами аналізу продуктивності й доступності без критичних зауважень, усі форми коректно відбивають спроби зловживання, тести проходять із цільовим покриттям, рішення про кешування задокументоване й не потребує архітектурних змін при майбутньому впровадженні.

## Stage 7 — Деплой на PythonAnywhere Free

**Мета:** продукт живий у реальному інтернеті.

**Орієнтовна тривалість:** ~1 тиждень.

### Задачі

- [ ] Створення облікового запису на хостингу, отримання коду проєкту на сервер
- [ ] Налаштування ізольованого середовища виконання, встановлення залежностей
- [ ] Налаштування змінних середовища через захищений механізм платформи
- [ ] Конфігурація точки входу веб-застосунку з посиланням на продакшн-налаштування
- [ ] Застосування міграцій, створення адміністративного облікового запису
- [ ] Збирання статичних файлів, компіляція перекладів
- [ ] Завантаження верифікованого контенту зі Stage 5 на продакшн-базу даних
- [ ] Підключення сервісу моніторингу помилок
- [ ] Повний прогін чек-листа перед запуском (розділ 18.5 ТЗ)
- [ ] Перша ручна процедура резервного копіювання бази даних і перевірка, що відновлення з цього знімка справді працює
- [ ] Прив'язка домену (за наявності), перевірка захищеного з'єднання
- [ ] Оновлення налаштувань конвеєра безперервної інтеграції (Stage 0) для автоматизованого розгортання на продакшн-середовище за успішного проходження перевірок, якщо технічно доцільно на цьому тарифі хостингу; інакше — фіксація ручної процедури оновлення коду

### Критерій готовності

Сайт відкривається за публічною адресою, показує коректний орнамент дня, реєстрація й вхід працюють, перший ручний бекап зроблено і перевірено на відновлюваність, чек-лист із розділу 18.5 ТЗ пройдений повністю.

## Stage 8 — Пост-реліз рутина

**Мета:** підтримка, зростання й ітеративне вдосконалення після запуску.

**Триває безкінечно.**

### Регулярні задачі

- [ ] Періодична перевірка, що знімок бекапу дійсно розгортається в чистому тестовому середовищі
- [ ] Моніторинг лічильника патернів зі статусом «помилка генерації» через дашборд
- [ ] Поступове розширення довідника регіонів і мотивів понад мінімальний набір зі Stage 5

### Стратегічні задачі (за готовності)

- [ ] Окремий документ SEO-стратегії
- [ ] Розгляд беклогу майбутніх можливостей (розділ 22 ТЗ) за пріоритетом
