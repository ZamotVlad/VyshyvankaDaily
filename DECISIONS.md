# Частина 1 — Архітектурні рішення (ADR)

## ADR 1 — Стратегія генерації URL (slug)

- **Дата:** 08.07.2026
- **Статус:** Прийнято
- **Контекст:** Необхідно обрати підхід до формування людино-зрозумілих URL для регіонів, статей блогу та категорій блогу. На відміну від типової UGC-платформи, увесь контент цього проєкту курований (немає масового автоматичного потоку записів від сторонніх користувачів), тому доступний ширший вибір стратегій, ніж у проєктах з UGC.
- **Рішення:** Для всіх slug-сутностей (`Region`, `BlogPost`, `BlogCategory`) — ручне введення осмисленого англійського slug адміністратором як основний спосіб (наприклад `/regions/poltava-region/`, `/blog/embroidery-colors-meaning/`), з автоматичною транслітерацією як розумний фолбек, якщо slug не заданий вручну. Оскільки весь контент курований, немає потреби у відмові від ручного контролю заради стабільності масового автоматичного створення — можна дозволити собі якість над швидкістю для кожного окремого запису.
  **Виняток:** `DailyPattern` не використовує текстовий slug взагалі — ідентифікується датою в URL (`/pattern/2026-07-08/`), оскільки сутність за визначенням прив'язана до календарної дати, а не довільного заголовка.
- **Наслідки:**
  - (+) Повний контроль над ключовими словами в URL для SEO-критичного контенту (регіони, блог).
  - (+) Простіша модель, ніж гібрид SetupHunt — не потрібно розділяти логіку на «курований контент» і «UGC», бо UGC тут немає взагалі.
  - (−) Адміністратор має пам'ятати вручну заповнювати slug для найкращого результату; фолбек транслітерації рятує від помилки, але дає менш «гарний» URL, якщо slug забуто.

## ADR 2 — Вибір бази даних: SQLite замість PostgreSQL на старті

- **Дата:** 08.07.2026
- **Статус:** Прийнято
- **Контекст:** У попередньому проєкті (SetupHunt) було свідомо обрано PostgreSQL з першого дня через потребу в `select_for_update()` для захисту голосування від одночасного запису (SQLite ігнорує row-level locks). У VyshyvankaDaily немає голосування, немає UGC, і єдина потенційно конкурентна точка запису — створення `DailyPattern` першим відвідувачем дня, що вже захищене звичайним `UniqueConstraint` на полі дати (працює однаково надійно в обох СУБД). Проєкт переважно читає, пише вкрай рідко.
- **Рішення:** SQLite як основна база даних на етапі PythonAnywhere Free. Перехід на PostgreSQL відкладається до появи реальної потреби (значне зростання паралельних записів, не читань — малоймовірний сценарій для цього типу продукту).
- **Наслідки:**
  - (+) Нуль додаткової інфраструктури на старті — не потрібен окремий сервер бази даних.
  - (+) Технічно обґрунтований вибір, не компроміс заради лінощів — навантаження проєкту дійсно read-heavy.
  - (−) Якщо в майбутньому з'явиться функціонал з інтенсивним паралельним записом (малоймовірно, але теоретично можливо при значному зростанні аудиторії) — знадобиться міграція на PostgreSQL; код спроєктовано так, щоб рядок підключення до бази даних абстрагований і зміна не вимагає переписування логіки.

## ADR 3 — Лінива генерація замість планувальника задач

- **Дата:** 08.07.2026
- **Статус:** Прийнято
- **Контекст:** Безкоштовний тариф цільового хостингу (PythonAnywhere Beginner) не надає можливості запускати задачі за розкладом (немає аналога cron у жодному вигляді на цьому тарифі). Традиційний підхід «генерувати контент наступного дня вночі через фоновий планувальник» технічно неможливий без переходу на платний тариф.
- **Рішення:** Контент дня генерується не за розкладом, а при першому запиті відвідувача до головної сторінки в цей календарний день («лінива генерація»). Контролер перевіряє наявність запису `DailyPattern` на сьогодні; за відсутності — синхронно викликає сервіс генерації в межах того самого HTTP-запиту, зберігає результат, повертає його. Кожен наступний відвідувач того самого дня отримує вже збережений результат.
- **Наслідки:**
  - (+) Функціонал повністю сумісний з обмеженнями безкоштовного тарифу, не вимагає жодного планувальника.
  - (+) Генерація відбувається щонайбільше раз на добу — мінімальне додаткове навантаження на обмежений добовий ліміт обчислювального часу.
  - (+) Рішення природно масштабується — при переході на платний тариф чи власний сервер можна додатково підстрахуватися завчасною генерацією через реальний плановий процес, без переробки самої логіки лінивої генерації (вона продовжує коректно працювати паралельно).
  - (−) Найперший відвідувач кожного нового дня отримує відповідь на частку секунди повільніше за наступних (через одноразову генерацію в межах його запиту) — прийнятний компроміс з огляду на легкість самої обчислювальної операції.

## 🏁 Stage 0: Фундамент, інфраструктура і якість коду — ЗАВЕРШЕНО
**Дата закриття:** 10.07.2026

### Що зроблено й перевірено:
1. Git-репозиторій, `.gitignore`, venv (Python 3.12).
2. `requirements/` розбито на `base`/`local`/`production`; **Django 5.2 LTS** (не 5.0/5.1 — обидва без security-патчів на момент старту).
3. Ruff + pre-commit (`ruff-check`/`ruff-format`, `rev: v0.15.20` — актуальна назва хука в новіших версіях `ruff-pre-commit`).
4. Конфігураційний пакет `config` розбитий на `settings/base|local|test|production`; `DEBUG`/`SECRET_KEY`/`ALLOWED_HOSTS` читаються через `django-environ`, без хардкоду.
5. `BASE_DIR` з трьома `.parent` (файл лежить у `config/settings/base.py`).
6. 5 доменних застосунків у `apps/` (`core`, `accounts`, `patterns`, `blog`, `pages`), зареєстровані з префіксом `apps.` в `INSTALLED_APPS` і в `name` кожного `apps.py`.
7. Абстрактні моделі `core` (`TimeStampedModel`, `SlugModel` з явним `allow_unicode=False`, розділ 3.1 ТЗ), `translation.py`-заглушка.
8. `django-modeltranslation` з нульового дня; `LANGUAGE_CODE="uk"`, `LANGUAGES=[uk, en]`, `LocaleMiddleware`.
9. Локалізація: `locale/uk`, `locale/en` згенеровані й скомпільовані (gettext через Docker-образ — на Windows-хості системний gettext ставити не знадобилось).
10. Docker: `Dockerfile` (з системним `gettext`), `docker-compose.yml`, `.dockerignore`.
11. CI (`.github/workflows/ci.yml`): `ruff`/`pre-commit` додані в `requirements/local.txt`, щоб CI міг їх знайти; фіктивний `SECRET_KEY` через `env:` у кроці `pytest`, бо `.env` навмисно не в git.
12. `pytest` налаштований на `config.settings.test` (MD5-хешер паролів для швидкості — розділ 21.2 ТЗ).
13. Фінальна перевірка: `manage.py check`, `migrate`, `showmigrations` (усі 5 застосунків зареєстровані), `docker compose up` — сайт піднявся й показав стандартну сторінку Django 5.2.
14. CI на GitHub Actions — зелений після фінального пуша.

### На що звернути увагу далі:
- Локальний порт для Docker — **9000:8000**, не 8000/8080: Windows Hyper-V резервує діапазони `7969–8168` і сусідні під NAT, тому стандартний 8000 недоступний саме на цій машині.
- `django.contrib.sites` ще не доданий — потрібен перед Stage 3 (Allauth).
- `STATICFILES_DIRS` вказує на `static/` — створена порожньою, реального вмісту ще немає.
- `LocaleMiddleware` доданий заздалегідь, свідомо, не через недогляд: Roadmap Stage 0 буквально вимагає лише підключення `django-modeltranslation`, без middleware — але оскільки проєкт двомовний за задумом ТЗ з першого дня, а `LocaleMiddleware` не залежить від наявності реальних моделей чи шаблонів, доданий одразу, щоб не переписувати `MIDDLEWARE` і не ризикувати порядком middleware вдруге в Stage 2.
- `apps/core/tests.py` містить лише заглушку `test_dummy_for_ci()` (`assert True`) — потрібна виключно щоб CI (`pytest`) не падав з Exit code 5 через відсутність тестів на порожньому проєкті. **Прибрати цю заглушку одразу, як тільки в Stage 1 з'явиться перша реальна модель** (`Region` або `SlugModel`-нащадок) — замінити на справжній тест транслітерації (кирилична назва → латинський slug) чи логіки симетрії генератора, залежно від того що буде готове раніше.
